Sistem Rekomendasi Playlist Musik Berbasis Konteks Aktivitas Pengguna Menggunakan Metode EDAS


4.1 Pendahuluan Implementasi

Bab ini menjelaskan implementasi sistem secara rinci mulai dari tahap pengolahan data, pemodelan preferensi pengguna, perhitungan metode EDAS, pembentukan playlist berbasis target durasi, integrasi antarmuka web, sampai integrasi layanan pihak ketiga (Spotify, YouTube, dan Gemini untuk narasi). Fokus utama penulisan bab ini adalah menunjukkan jejak yang jelas antara rancangan metodologi pada bab sebelumnya dengan realisasi di kode program.

Implementasi sistem disusun dengan prinsip traceability, yaitu setiap keputusan metodologis memiliki representasi yang eksplisit pada fungsi, modul, dan endpoint yang berjalan. Dengan prinsip ini, sistem tidak hanya bekerja secara fungsional, tetapi juga mudah diaudit untuk kebutuhan akademik: penguji dapat menelusuri bagaimana data dibaca, bagaimana bobot dibentuk, bagaimana skor EDAS dihitung, dan bagaimana hasil akhir ditransformasikan menjadi playlist yang dapat diekspor.

Selain itu, implementasi dilakukan dengan pemisahan concern yang tegas. Lapisan komputasi (preprocessing, preferensi, EDAS, dan playlist builder) ditempatkan terpisah dari lapisan presentasi (UI) serta lapisan integrasi layanan eksternal (OAuth dan API platform musik). Pemisahan ini penting karena menjaga agar kontribusi ilmiah utama penelitian—yaitu mekanisme rekomendasi berbasis EDAS—tetap dapat dievaluasi secara independen dari komponen antarmuka dan infrastruktur.


4.2 Gambaran Arsitektur Implementasi

Secara implementatif, arsitektur dibagi menjadi dua area kerja:
(1) Backend komputasi penelitian di folder `backend/src`, dan
(2) Frontend + API adapter di folder `frontend/src`.

Pada sisi backend komputasi, file utama terdiri dari:
- `backend/src/load_data.py` untuk pembacaan dataset,
- `backend/src/preprocess.py` untuk pembersihan data dan pembentukan decision matrix,
- `backend/src/preferences.py` untuk konversi kuesioner menjadi bobot dan tipe kriteria,
- `backend/src/edas.py` untuk kalkulasi EDAS,
- `backend/src/playlist_builder.py` untuk penyusunan playlist berbasis durasi,
- `backend/src/main.py` untuk orkestrasi pipeline.

Pada sisi frontend/API, arsitektur menggunakan pola route adapter tipis:
- Endpoint tetap berada di `frontend/src/app/api/**`,
- Logika domain dipindahkan ke `frontend/src/server/modules/**`,
- Utility bersama dipusatkan pada `frontend/src/server/utils/**` dan `frontend/src/lib/**`.

Dengan pola ini, endpoint publik tidak berubah, tetapi struktur kode menjadi lebih modular dan lebih mudah didokumentasikan di bab implementasi.


4.3 Implementasi Inti Sistem

4.3.1 Implementasi Data dan Pemodelan Kriteria

Subbab ini menjelaskan bagaimana data mentah dari CSV diubah menjadi data siap hitung untuk EDAS, serta bagaimana jawaban kuesioner dikonversi menjadi parameter preferensi terukur. Pada fase ini, akurasi data menjadi prioritas utama karena kualitas hasil rekomendasi sangat ditentukan oleh kualitas input.

Tahap implementasi dimulai dari validasi keberadaan file, pembacaan data tabular, validasi kolom wajib, normalisasi tipe data numerik, dan normalisasi durasi. Setelah data valid, sistem membentuk decision matrix dengan delapan kriteria audio sebagaimana dirumuskan di metodologi penelitian: `bpm`, `energy`, `danceability`, `happiness`, `popularity`, `acousticness`, `instrumentalness`, dan `speechiness`.

Selanjutnya, preferensi pengguna tidak diperlakukan sebagai data bebas, melainkan dipetakan ke model kriteria yang konsisten. Empat belas butir kuesioner dipetakan ke delapan parameter audio dengan dukungan reverse scoring pada butir tertentu. Hasil agregasi parameter diubah menjadi bobot ter-normalisasi dan label tipe kriteria (`benefit`, `cost`, `neutral`) untuk kebutuhan formula EDAS.

C. Implementasi Normalisasi Durasi

Snippet 4.3.1-3 (`backend/src/preprocess.py`):
```python
def parse_duration_to_seconds(duration_text: str) -> int:
    value = str(duration_text).strip()
    parts = value.split(":")

    if len(parts) != 2:
        raise ValueError(f"Format durasi tidak valid: {duration_text}")

    minutes_text, seconds_text = parts
    minutes = int(minutes_text)
    seconds = int(seconds_text)

    if minutes < 0 or seconds < 0:
        raise ValueError(f"Nilai durasi tidak valid: {duration_text}")

    normalized_minutes = minutes + (seconds // 60)
    normalized_seconds = seconds % 60

    return normalized_minutes * 60 + normalized_seconds
```

Snippet 4.3.1-4 (`backend/src/preprocess.py`):
```python
def _safe_parse_duration_to_seconds(duration_text: str) -> int | None:
    try:
        return parse_duration_to_seconds(duration_text)
    except (TypeError, ValueError):
        return None
```

Narasi:
Implementasi durasi mengantisipasi data scraping yang tidak selalu ideal, misalnya format anomali seperti `2:60`. Dengan normalisasi menit-detik, kasus seperti itu tetap diproses menjadi nilai yang valid (`180` detik) alih-alih langsung menggagalkan seluruh pipeline. Wrapper aman (`_safe_parse_duration_to_seconds`) membuat baris invalid dapat dieliminasi terkontrol di tahap `dropna` tanpa menghentikan proses keseluruhan.

```python
def _validate_answers(answers: dict[int, int]) -> None:
    expected_ids = {item.question_id for item in QUESTIONNAIRE_MAPPING}
    given_ids = set(answers.keys())

    missing = sorted(expected_ids - given_ids)
    extra = sorted(given_ids - expected_ids)

    if missing:
        raise ValueError(f"Jawaban belum lengkap. Nomor yang belum diisi: {missing}")

    if extra:
        raise ValueError(f"Terdapat nomor pertanyaan tidak valid: {extra}")

    for question_id, answer in answers.items():
        if answer < 1 or answer > 5:
            raise ValueError(f"Skala Likert harus 1-5. Nilai invalid pada nomor {question_id}: {answer}")
```

Narasi:
Validasi jawaban dibuat ketat karena bobot kriteria sangat sensitif terhadap input. Implementasi memastikan tiga hal sekaligus: tidak ada item yang kosong, tidak ada nomor liar di luar kuesioner, dan tidak ada nilai skala di luar rentang Likert. Dengan mekanisme ini, bobot yang dihasilkan dapat dipertanggungjawabkan secara metodologis.

4.3.2 Implementasi Perhitungan Metode EDAS

Pada subbab ini, implementasi formula EDAS dijelaskan dari tahap inisialisasi hingga ranking akhir. Semua komponen matematis yang dirancang pada metodologi diterjemahkan ke operasi DataFrame/Series agar perhitungan efisien dan transparan.

Implementasi didesain defensif terhadap kasus tepi, khususnya pembagian dengan nol pada nilai rata-rata atau nilai maksimum. Pendekatan ini tidak mengubah konsep metode, namun menjaga kestabilan komputasi agar sistem tetap menghasilkan keluaran valid pada distribusi data yang ekstrem.

Narasi:
Rata-rata dihitung per kolom kriteria (`axis=0`). Nilai ini menjadi baseline pembanding setiap alternatif, sesuai rumus AV pada metode EDAS.

E. Implementasi PDA dan NDA untuk Benefit/Cost

Snippet 4.3.2-5 (`backend/src/edas.py`):
```python
pda = pd.DataFrame(0.0, index=matrix.index, columns=criteria)
nda = pd.DataFrame(0.0, index=matrix.index, columns=criteria)

for criterion in criteria:
    av = average_solution[criterion]
    values = matrix[criterion]
    criterion_type = str(criteria_types[criterion]).lower().strip()

    if criterion_type == "cost":
        pda_column = (av - values).clip(lower=0)
        nda_column = (values - av).clip(lower=0)
    else:
        pda_column = (values - av).clip(lower=0)
        nda_column = (av - values).clip(lower=0)

    if av == 0:
        pda[criterion] = 0.0
        nda[criterion] = 0.0
    else:
        pda[criterion] = pda_column / av
        nda[criterion] = nda_column / av
```

Narasi:
Implementasi di atas adalah inti diferensiasi benefit vs cost. Untuk kriteria cost, deviasi positif dihitung ketika nilai alternatif lebih kecil dari AV. Untuk kriteria benefit, deviasi positif dihitung ketika nilai alternatif lebih besar dari AV. Operasi `clip(lower=0)` mewakili operator `max(0, x)` dalam rumus.

F. Implementasi SP dan SN Berbobot

Snippet 4.3.2-6 (`backend/src/edas.py`):
```python
weighted_pda = pda.mul(weights_series, axis=1)
weighted_nda = nda.mul(weights_series, axis=1)

sp = weighted_pda.sum(axis=1)
```

Narasi:
Setiap nilai deviasi dikalikan bobot kriterianya, lalu dijumlah per alternatif (per baris). Inilah tahap agregasi multi-kriteria yang mengubah informasi per-parameter menjadi skor komposit.

G. Implementasi NSP, NSN, dan Appraisal Score

Snippet 4.3.2-7 (`backend/src/edas.py`):
```python
max_sp = float(sp.max())
max_sn = float(sn.max())

if max_sp == 0:
    nsp = pd.Series(0.0, index=sp.index)
else:
    nsp = sp / max_sp

if max_sn == 0:
    nsn = pd.Series(1.0, index=sn.index)
else:
    nsn = 1 - (sn / max_sn)

appraisal_score = 0.5 * (nsp + nsn)
ranking = appraisal_score.rank(ascending=False, method="min").astype(int)
```

Narasi:
Normalisasi dilakukan agar komponen positif (NSP) dan komponen negatif (NSN) berada pada skala yang sebanding sebelum digabungkan menjadi appraisal score. Penentuan `rank` menggunakan urutan menurun (`ascending=False`) agar alternatif dengan skor tertinggi mendapat peringkat terbaik.

4.3.3 Implementasi Pembentukan Playlist Berbasis Durasi

Setelah ranking alternatif dihasilkan, sistem menerapkan strategi seleksi berbasis target durasi. Strategi ini penting karena konteks penggunaan playlist umumnya terkait batas waktu aktivitas pengguna (misalnya sesi belajar, perjalanan, atau olahraga).

Implementasi dilakukan dengan mekanisme greedy berdasarkan urutan appraisal score tertinggi. Lagu ditambahkan satu per satu sampai durasi kumulatif mencapai atau sedikit melampaui target. Pendekatan ini dipilih karena sederhana, deterministik, dan sesuai kebutuhan praktis aplikasi.

Narasi:
`PlaylistBuildResult` tidak hanya menyimpan daftar lagu, tetapi juga metrik evaluasi durasi. Ini memudahkan pembahasan di skripsi mengenai seberapa dekat hasil playlist terhadap target yang diminta pengguna.

B. Implementasi Seleksi Lagu Berbasis Durasi

Snippet 4.3.3-2 (`backend/src/playlist_builder.py`):
```python
def build_playlist_by_duration(
    ranked_songs: pd.DataFrame,
    target_duration_minutes: int,
) -> PlaylistBuildResult:
  if target_duration_minutes <= 0:
    empty_df = pd.DataFrame(columns=list(ranked_songs.columns) + ["playlist_order", "cumulative_duration_seconds"])
    return PlaylistBuildResult(
      playlist_df=empty_df,
      total_duration_seconds=0,
      target_duration_seconds=0,
      remaining_seconds=0,
      exceeded_seconds=0,
    )

  required_columns = {"duration_seconds", "appraisal_score", "rank"}
  missing_columns = sorted(required_columns - set(ranked_songs.columns))
  if missing_columns:
    raise ValueError(f"Kolom wajib untuk pembentukan playlist belum tersedia: {missing_columns}")

    target_duration_seconds = int(target_duration_minutes * 60)

    ordered_df = ranked_songs.sort_values(
        by=["appraisal_score", "rank"],
        ascending=[False, True],
    ).reset_index(drop=True)

    selected_rows: list[dict] = []
    total_duration_seconds = 0

    for _, row in ordered_df.iterrows():
        row_duration = int(row["duration_seconds"])
        row_dict = row.to_dict()

        total_duration_seconds += row_duration
        row_dict["playlist_order"] = len(selected_rows) + 1
        row_dict["cumulative_duration_seconds"] = total_duration_seconds
        selected_rows.append(row_dict)

        if total_duration_seconds >= target_duration_seconds:
            break
```

Narasi:
Urutan ranking dipakai sebagai prioritas utama. Ketika lagu ditambahkan, sistem sekaligus menyimpan urutan playlist (`playlist_order`) dan durasi kumulatif agar proses seleksi dapat diaudit langkah demi langkah.

C. Implementasi Metrik Selisih Durasi

```typescript
async function loadCsvText(): Promise<{ csvText: string; source: string }> {
  const localCandidates = [
    path.resolve(process.cwd(), "..", "data", "output.csv"),
    path.resolve(process.cwd(), "data", "output.csv"),
    path.resolve(process.cwd(), "public", "output.csv"),
  ];

  for (const candidate of localCandidates) {
    try {
      const csvText = await readFile(candidate, "utf-8");
      if (csvText.trim().length > 0) {
        return { csvText, source: candidate };
      }
    } catch {
      continue;
    }
  }

  const remoteUrl = process.env.DUMMY_PLAYLIST_CSV_URL ?? DEFAULT_REMOTE_CSV_URL;
  const response = await fetch(remoteUrl, { cache: "no-store" });

  if (!response.ok) {
    throw new Error(`Tidak bisa membaca CSV dari lokal maupun remote (${remoteUrl}).`);
  }

  const csvText = await response.text();
  if (csvText.trim().length === 0) {
    throw new Error(`CSV remote kosong (${remoteUrl}).`);
  }

  return { csvText, source: remoteUrl };
}
```

```typescript
function buildPlaylistFromCsv(tracks: CsvTrack[], targetMinutes: number): PlaylistItem[] {
  const shuffledTracks = shuffleTracks(tracks);
  const targetSec = Math.max(15, targetMinutes) * 60;
  const maxItems = Math.min(50, shuffledTracks.length);
  const items: PlaylistItem[] = [];
  let totalSec = 0;
  let index = 0;

  while (index < maxItems && totalSec < targetSec) {
    const track = shuffledTracks[index];
    const appraisalScore = Number((Math.max(0, track.popularity) / 100).toFixed(4));

    items.push({
      rank: index + 1,
      title: track.title,
      artist: track.artist,
      durationSec: track.durationSec,
      appraisalScore,
    });

    totalSec += track.durationSec;
    index += 1;
  }

  return items;
}
```

4.4 Validasi Implementasi

4.4.1 Validasi Struktural dan Arsitektural

Validasi pertama dilakukan pada struktur implementasi. Endpoint publik tetap dipertahankan pada jalur semula, sementara logika utama dipindahkan ke modul server (`server/modules`). Hasilnya, kontrak API yang dikonsumsi UI tidak berubah, tetapi maintainability meningkat karena logika bisnis terpusat.

Validasi kedua adalah traceability terhadap metodologi. Formula inti EDAS dapat ditelusuri langsung di `backend/src/edas.py`, pembentukan bobot dan tipe kriteria ada di `backend/src/preferences.py`, sedangkan orkestrasi komputasi terdapat di `backend/src/main.py`. Ini membuktikan bahwa implementasi tidak menyimpang dari kerangka penelitian.

4.4.3 Validasi Fungsional End-to-End (Naratif)

Dari sisi alur pengguna, validasi fungsional menunjukkan bahwa:
1) pengguna dapat melewati alur konteks -> proses -> hasil,
2) playlist tampil dengan metrik durasi dan skor,
3) status koneksi Spotify/YouTube dapat dibaca,
4) proses ekspor memberikan umpan balik kuantitatif (`totalAdded`, `totalMissing`), dan
5) NLG menampilkan narasi berbasis konteks, dengan fallback jika layanan model gagal.

Dengan hasil tersebut, implementasi telah memenuhi kebutuhan fungsional penelitian sekaligus kebutuhan presentasi hasil pada antarmuka.


4.5 Keterkaitan Implementasi dengan Tujuan Penelitian

Tujuan utama penelitian adalah menghasilkan rekomendasi playlist berbasis konteks aktivitas pengguna menggunakan metode EDAS. Pada implementasi, tujuan tersebut terwujud melalui rangkaian pipeline yang jelas: data dibersihkan, preferensi dikonversi menjadi parameter keputusan, skor EDAS dihitung, lalu alternatif terbaik disusun menjadi playlist berdasarkan target durasi.

Komponen tambahan seperti ekspor platform dan NLG tidak mengubah inti keputusan metode. Keduanya berperan sebagai lapisan utilitas: ekspor membuat hasil rekomendasi dapat langsung digunakan di platform riil, sedangkan NLG meningkatkan keterbacaan hasil bagi pengguna non-teknis.

Secara keseluruhan, implementasi telah menunjukkan konsistensi antara teori dan praktik. Rumus-rumus EDAS tidak berhenti pada penjelasan konseptual, tetapi benar-benar dijalankan pada data nyata, menghasilkan ranking, dan diwujudkan menjadi produk fungsional yang dapat diuji dari hulu ke hilir.
